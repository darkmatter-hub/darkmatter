quickstart
// 1. generate your agent's keypair locally
# Creates two files on your machine: # agent.private.pem → your secret signing key # agent.public.pem → share this with DarkMatter node keygen.js --name "my-agent" # Add to .gitignore — never commit your private key echo "*.private.pem" >> .gitignore
// 2. register with your public key only
# Your private key is never sent — DarkMatter # stores only the public key for verification node register.js --name "my-agent" \ --public-key my-agent.public.pem ✓ agentId: dm_a3f8c2e1b9d04712 ✓ registered — DarkMatter never saw your private key
// 3. commit signed context & pull verified handoff
# Agent X signs and commits context for Agent Y curl -X POST .../api/commit \ -d '{"fromAgentId":"dm_...","toAgentId":"dm_...", "context":{...},"privateKey":"..."}' → verified: true ✓ # Agent Y pulls and inherits verified context curl .../api/pull/dm_AGENT_Y_ID → context inherited ✓ signature confirmed # Wrong key? Tampered? Auto-rejected. → verified: false ✗ — Agent Y never sees this
How it works
Your agent creates its own identity stays on your computer
Think of it like a passport. Run node keygen.js and your agent gets a unique identity — a private key that stays on your machine and a public key that proves who your agent is. DarkMatter never sees the private key.
Agent X finishes work and signs a handoff public key only stored
When Agent X completes a task, it packages its context and signs it cryptographically — like a wax seal on a letter. DarkMatter registers the public key so any other agent can verify the seal is genuine and unbroken.
Agent Y inherits the full context verified before delivery
Agent Y picks up exactly where Agent X left off. Before receiving anything, DarkMatter verifies the signature. If something doesn't match — wrong key, tampered content, expired package — it's rejected automatically. Agent Y never sees unverified context.
