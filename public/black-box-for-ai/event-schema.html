Black Box
Event Schema

The Black Box for AI definition specifies that a conforming system must record inputs, outputs, timestamps, agent identity, event sequence, and integrity proofs. This document defines the typed event taxonomy that turns those abstract requirements into a concrete, testable protocol.

Every action an AI agent takes belongs to one of these event types. Typed events are what make a record reconstructable — not just a sequence of blobs, but a traceable chain of classified actions where investigators can ask "show me every decision event" or "show me every human override" and get a clean answer.

01 — Base event schema

Every event shares this structure

Every event recorded by a Black Box for AI — regardless of type — must contain the base fields below. These are the fields that satisfy the six required properties in the canonical definition. Event-type-specific fields extend this base; they do not replace it.

Base event — required fields Schema v1.0
{
  "event_id": "evt_01H...", // unique identifier — immutable once assigned
  "event_type": "decision", // see event taxonomy below
  "trace_id": "trc_01H...", // groups events into one pipeline run
  "agent": {
    "id": "dm_abc123", // authenticated agent identity
    "model": "claude-sonnet-4-6", // model and version — strongly recommended
    "provider": "anthropic"
  },
  "input": { ... }, // what the agent received — required
  "output": { ... }, // what the agent produced — required
  "timestamps": {
    "client_time": "2026-04-15T16:21:09Z", // agent-asserted — part of signed envelope
    "accepted_time": "2026-04-15T16:21:10Z" // server ledger — independent of agent
  },
  "sequence": {
    "parent_event_id": "evt_previous...", // causal chain linkage
    "position": 3 // position in log — immutable
  },
  "integrity": {
    "payload_hash": "sha256:a9c4e1...", // computed client-side before transmission
    "parent_hash": "sha256:7f3a9b...", // links to previous event's integrity hash
    "agent_signature": "ed25519:..." // signed by agent before transmission
  }
}
Field Type Status Definition requirement satisfied
event_id string Required Unique, stable identifier for this event. Immutable once assigned.
event_type enum Required Classifies the action. Enables post-incident filtering by action category.
agent.id string Required Satisfies agent identity. Must be authenticated, not self-reported.
input object Required Satisfies inputs. What the agent received before acting.
output object Required Satisfies outputs. What the agent produced.
timestamps.client_time ISO 8601 Required Satisfies timestamp. Agent-asserted time, part of signed envelope.
timestamps.accepted_time ISO 8601 Required Independent server receipt time. Separate from client-asserted time.
sequence.parent_event_id string Required Satisfies event sequence. Links to predecessor event in chain.
integrity.payload_hash string Required Satisfies integrity proof. SHA-256 of canonical payload, computed before transmission.
integrity.agent_signature string Required Ed25519 signature over canonical envelope. Authenticates agent identity and chain position.
agent.model string Recommended Model and version. Typically the first field needed in post-incident reconstruction.
trace_id string Recommended Groups events into a single pipeline run. Required for multi-agent trace reconstruction.

02 — Agent execution events

Actions the agent takes autonomously

These are the core events that record what the agent did. Every pipeline action belongs to one of these types. Decision and tool_call are the most important — they cover the two categories where agent behavior is most consequential and most likely to be investigated.

Agent execution event types

decision Required Agent evaluated options and chose a course of action
The primary event type. Use whenever the agent produces a meaningful output in response to a prompt or context — approving, rejecting, classifying, generating, routing. If you only record one event type, record this one.
input: { prompt, context }  |  output: { decision, reasoning, confidence }
tool_call Required Agent invoked an external tool, API, or function
Records every external action the agent took — API calls, database queries, web searches, code execution. Tool calls are where agent behavior becomes operationally consequential. Most real failures trace back to a tool call, not the final output. Recording these as first-class events is the difference between a record and a black box.
input: { tool_name, parameters }  |  output: { result, status_code, latency_ms }
message Recommended Agent sent or received a message in a conversation
Captures conversational context — user messages, agent responses, system prompts. Use when the pipeline involves dialogue rather than discrete task execution.
input: { role, content }  |  output: { role, content }
handoff Recommended Agent passed context to another agent
Records multi-agent transitions — when Agent A finishes and passes work to Agent B. Creates the cross-agent lineage chain that makes multi-agent pipelines fully reconstructable.
input: { context, from_agent_id }  |  output: { to_agent_id, payload_hash }
checkpoint Recommended Agent saved mid-task state without completing a handoff
Progress save during a long-running task. Not a final output — a recoverable state. Enables reconstruction of where a task was at any point, not just at the beginning and end.
input: { task_description, progress }  |  output: { state_snapshot }
error Recommended Agent failed mid-task, logging last known state
Records failure state — the last known context before the agent stopped. This is the most important event when the investigation starts. A black box that doesn't record failures is least useful precisely when it is most needed.
input: { last_context }  |  output: { error_type, error_message, stack_trace }

03 — Human interaction events

Moments where a human entered the loop

These events record the moments that matter most in disputes. If an AI system made a consequential decision and a human later claims they reviewed it, approved it, or changed it — the black box record of that interaction is the only independent evidence of what actually happened.

Human interaction event types

human_approval Recommended Human explicitly approved an agent action before execution
EU AI Act Article 14 requires human oversight for high-risk AI systems. Recording approvals as first-class events is how you demonstrate that requirement was met — not by assertion, but by an independent record.
input: { agent_proposal }  |  output: { approved_by, approved_at, conditions }
human_override Recommended Human changed or rejected an agent's output
Records when a human corrected what the agent produced. The original agent output and the human correction both live in the record, with the identity of the reviewer. Critical for any investigation that involves "the AI did this but a human changed it."
input: { original_output }  |  output: { corrected_output, reviewer_id, reason }
human_escalation Recommended Agent paused and flagged for human review
Records the moment an agent stopped and waited for human input. Creates a verifiable record that the system did not proceed autonomously past a defined threshold — which matters for both compliance and liability.
input: { reason_for_escalation }  |  output: { escalated_to, escalation_context }
consent Recommended Human explicitly consented to a proposed agent action
Distinct from approval — consent records that the affected party (not just a reviewer) agreed to what the agent was about to do. Relevant in regulated industries where patient, customer, or user consent must be documented.
input: { proposed_action, consent_request }  |  output: { consented_by, consent_scope }

04 — System and lifecycle events

Pipeline structure and operational state

These events record the structure of the pipeline itself — when agents are spawned, when pipelines branch, when tasks time out. They are what make a reconstruction complete rather than just a sequence of decisions.

System event types

agent_spawn Recommended Agent dynamically created a child agent
Records multi-agent hierarchy. When an orchestrator creates sub-agents, the spawn event creates the lineage link — making the full tree reconstructable, not just the leaf-level actions.
input: { spawn_reason, parent_agent_id }  |  output: { child_agent_id, initial_context }
branch Recommended Pipeline split into parallel execution paths
Records fork points where a pipeline runs multiple paths in parallel. Without this event, parallel execution paths appear as disconnected chains. With it, the full branching structure is reconstructable.
input: { branch_context }  |  output: { branch_ids[], branch_reason }
merge Recommended Parallel branches rejoined into a single path
Records when parallel paths converge. Links the merge event to all contributing branch events, creating a complete DAG rather than a linear chain.
input: { contributing_branch_ids[] }  |  output: { merged_context }
timeout Recommended Agent hit deadline before completing its task
Records time-based failures. Important for any post-incident analysis where the question is "why did the pipeline stop?" without an explicit error.
input: { task_at_timeout, deadline }  |  output: { elapsed_ms, partial_output }
retry Recommended Agent reattempted a previously failed task
Links to the original failed event. Creates a record that distinguishes "the agent succeeded on the first attempt" from "the agent succeeded after three retries" — which is relevant for both debugging and compliance.
input: { original_event_id, retry_context }  |  output: { attempt_number, outcome }

05 — Compliance events

Regulatory and data governance actions

These events exist because regulatory frameworks require evidence of specific actions — not just general audit logs. They map directly to articles in the EU AI Act, GDPR, and sector-specific regulations. Recording them as first-class typed events means a compliance export can answer "show me all consent events from Q1" without parsing freeform text.

Compliance event types

policy_check Recommended System evaluated output against a defined policy
Records automated policy evaluations — content filters, safety checks, compliance rules. Creates a verifiable record that governance controls were applied, not just that they exist.
input: { output_evaluated, policy_id }  |  output: { passed, violations[], policy_version }
redact Recommended PII or sensitive data removed before handoff or storage
Records that data was removed and why — without storing the removed data. Maps to EU AI Act Article 10 and GDPR data minimisation requirements. The record proves redaction happened without recreating the redacted content.
input: { pre_redaction_hash }  |  output: { redaction_reason, fields_redacted[], post_redaction_hash }
audit_access Recommended External auditor or regulator accessed the record
Records third-party access to the audit trail itself. Creates a meta-record of who accessed the black box, when, and what they accessed — which matters for chain-of-custody in formal investigations.
input: { access_request }  |  output: { accessed_by, accessed_at, scope }

06 — Conformance and versioning

What it means to conform to this schema

A conforming implementation must record all Required base fields for every event. It must use one of the defined event types, or declare a custom type in the format custom.{namespace}.{type} to avoid collision with reserved types.

Recommended fields are not required for conformance, but their absence should be noted in any conformance claim. A claim of "conforms to Black Box Event Schema v1.0" without model attribution, tool calls, or human interaction events is technically valid — but incomplete for most real-world investigation scenarios.

This schema is versioned. Version 1.0 is stable. Future versions will add new event types and fields — they will not remove or rename existing ones. Any breaking change requires a new major version.
