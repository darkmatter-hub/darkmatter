Integrity model

Verification without trust

DarkMatter is an execution record you can verify without trusting DarkMatter. Your agent hashes each step client-side, signs the commit envelope with its own Ed25519 key, and submits it to an append-only log with Merkle inclusion proofs and signed checkpoints. Verification works offline — use the provided verifier or build your own from the published spec.

Core principle

The fundamental question DarkMatter answers is: when something goes wrong with an AI agent, can you prove what it actually did? Not what the logs say. Not what the operator claims. Prove it — cryptographically, to anyone, offline, without depending on DarkMatter to be honest.

Design target
Even if DarkMatter wanted to alter a record, any modification would be mathematically detectable by anyone holding a commit receipt or a signed checkpoint.

This is achieved by moving the trust burden from institutional promises to cryptographic proof. The system is designed so that DarkMatter's servers are a convenience, not a requirement for verification.

What is signed

Agents sign a commit envelope with their own Ed25519 private key before sending anything to DarkMatter. The private key never leaves the agent's machine.

The envelope contains:

{
  "schema_version":        "2",
  "agent_id":              "dm_abc123",
  "key_id":                "default",
  "timestamp":             "2026-04-09T12:00:00Z",   // client-asserted time
  "payload_hash":          "sha256 of canonical(payload)",
  "parent_integrity_hash": "sha256 of parent envelope | \"root\""
}

The agent signs canonical(envelope) — the canonical JSON serialization of this object. Signing the full envelope (not just the payload) means the agent authenticates: what was submitted, where in the chain it links, who submitted it, under which key, and at what time.

What this prevents
The operator cannot alter commit metadata (timestamp, chain position, agent identity) while keeping a valid agent signature. Any such alteration breaks the signature and is immediately detectable.

What is hashed

Payload hash

payload_hash = SHA-256( canonical(payload) )

Computed by the SDK before transmission. The server recomputes and flags any mismatch. DarkMatter cannot silently alter a payload without the client detecting it on the next read.

Integrity hash

integrity_hash = SHA-256( canonical(envelope) )

The integrity hash uniquely identifies a commit. Two commits with identical payloads and parents but different agents, keys, or timestamps produce different integrity hashes. This is what makes each commit cryptographically unique.

Canonical serialization

All hashing goes through canonical serialization first — a deterministic JSON encoding where object keys are sorted lexicographically, null values are kept, and floats follow a precisely specified rule. This ensures Python, TypeScript, and any future SDK produce identical byte sequences for the same data.

The canonical serialization spec is published in INTEGRITY_SPEC_V1.md and validated by 18 cross-language test vectors.

What is logged

Every accepted commit is appended to an append-only log table in Supabase. The table has no UPDATE or DELETE row-level security policies — inserts only. Each log entry records:

  • log_position — sequential integer, immutable once assigned
  • integrity_hash — the commit's integrity hash
  • leaf_hash — RFC 6962 Merkle leaf hash binding commit to its position
  • tree_root — Merkle root of all leaves 0..position at time of append
  • accepted_at — server ledger timestamp (distinct from client_timestamp)

Dual timestamps

Every commit carries two separate timestamps that are never conflated:

FieldSet byMeaningIn signed envelope
client_timestampAgentWhat the agent assertedYES
accepted_atServer ledgerWhen DarkMatter accepted the commitNO

The accepted_at timestamp is used in the Merkle leaf envelope, binding each commit to its position in server time. The client_timestamp is part of the signed surface — the agent authenticates what time they claimed when they submitted.

What is checkpointed

Every 10 minutes, DarkMatter signs the current Merkle tree root and publishes it to. This is a surface DarkMatter cannot unilaterally modify — GitHub's commit history makes past checkpoints tamper-evident.

A checkpoint contains:

{
  "schema_version":     "3",
  "checkpoint_id":      "cp_1775746920457_ce1b057f",
  "tree_root":          "c26618dd...",  // Merkle root of entire log
  "tree_size":          318,            // number of commits in tree
  "log_root":           "0bfa92fa...",  // running hash of all entries
  "log_position":       317,
  "timestamp":          "2026-04-09T15:20:54Z",
  "previous_cp_id":     "cp_...",       // chain linkage
  "previous_tree_root": "...",
  "server_sig":         "..."           // Ed25519 over canonical(above)
}

The checkpoint is signed over its canonical form — the same signing rules as commits. Anyone can verify the checkpoint signature using DarkMatter's public key at GET /api/log/pubkey.

Phase 1 — Client-side hashing + envelope signing

Complete

The SDK computes payload_hash and integrity_hash locally before transmission. The agent signs the commit envelope with its Ed25519 private key. The server validates all hashes and signatures — mismatches are flagged in the commit receipt.

What you can claim: "Commits are hashed and signed client-side. Any alteration to a payload or envelope is detectable offline, without contacting DarkMatter."

Phase 2 — Append-only log + signed checkpoints

Complete

Every commit is appended to the log. Every 10 minutes, DarkMatter signs the log root and publishes it to GitHub. The append-only log means DarkMatter cannot silently delete a commit — deletion changes the log root, making it detectable by anyone holding the previous checkpoint.

What you can claim: "DarkMatter cannot silently rewrite history. Any deletion or modification of a past commit changes the Merkle tree root, which would be inconsistent with the signed and published checkpoint."

Phase 3 — Merkle inclusion proofs

Complete

Every commit gets a Merkle inclusion proof at commit time, returned in the commit receipt. The proof is a set of sibling hashes (O(log n) in size) that lets anyone verify a specific commit is included in a signed tree root — without downloading the entire log.

Merkle leaf format

leaf_hash = SHA-256( 0x00 || UTF-8( canonical({
  "accepted_at":    "2026-04-09T15:00:00Z",
  "commit_id":      "ctx_...",
  "integrity_hash": "hex...",
  "log_position":   317
}) ) )

The 0x00 domain separation prefix (RFC 6962) prevents second-preimage attacks. Each leaf binds the commit to its exact position in the log — position forgery is detectable.

What you can claim: "You don't need to trust DarkMatter at all. Given any commit receipt and any published checkpoint, you can verify offline that the commit is included in the signed Merkle tree root — with no network calls."

Phase 4A — External witnesses

Live

Independent witnesses co-sign each checkpoint with their own Ed25519 keys. A witness verifies DarkMatter's server signature before co-signing — refusing to sign any checkpoint where DarkMatter's signature is invalid.

What a witness signature proves: "At timestamp T, [witness] independently confirmed that the DarkMatter log had Merkle tree root R at tree size N, and that DarkMatter's server signature on this checkpoint is cryptographically valid."

Witness signatures are included in export bundles and verifiable offline using only the witness public key — no DarkMatter servers required.

Current state
Phase 4A has one active witness node operated by the DarkMatter team on separate infrastructure. We are actively seeking independent organisations to operate witness nodes. Contact us to run a witness node.
What you can claim: "Each checkpoint is co-signed by an independently operated witness. Tampering with history requires compromising both DarkMatter's infrastructure and the witness infrastructure simultaneously."
What Phase 4A does NOT yet claim
Independent log reconstruction by witnesses. Multi-party consensus (2-of-3 quorum). Censorship resistance. Those are Phase 4B.

What a proof receipt contains

Every commit returns a _proof receipt containing everything needed to verify inclusion offline:

{
  "_proof": {
    "log_position":    317,
    "leaf_hash":       "hex...",
    "tree_root":       "hex...",
    "tree_size":       318,
    "accepted_at":     "2026-04-09T15:00:00Z",
    "client_timestamp": "2026-04-09T14:59:59Z",
    "inclusion_proof": {
      "leaf_index": 317,
      "tree_size":  318,
      "proof": [
        { "hash": "hex...", "direction": "left" },
        { "hash": "hex...", "direction": "right" }
      ]
    },
    "proof_status":    "checkpointed_witnessed",
    "verify_url":      "https://darkmatterhub.ai/api/log/proof/ctx_..."
  }
}

Offline verification

The standalone verifier works with zero DarkMatter dependencies. It is included in every export bundle.

# Verify chain structure only (Phase 1)
python verify_darkmatter_chain.py bundle.json

# Full verification — structure + checkpoint sig + Merkle proofs + witnesses
python verify_darkmatter_chain.py bundle.json \
  --checkpoint checkpoint.json \
  --pubkey server_pubkey.pem

# Check checkpoint consistency between two snapshots (Phase 3.5)
python verify_darkmatter_chain.py bundle.json \
  --checkpoint old_checkpoint.json \
  --checkpoint-b new_checkpoint.json \
  --pubkey server_pubkey.pem

# Run the full phase validation suite against a live system
python validate_phases.py \
  --api-key dm_sk_... \
  --base-url https://darkmatterhub.ai

Exit codes: 0 = verified, 1 = failed, 2 = malformed input.

Precise claims

We are deliberate about what we claim and what we don't. The following statements are accurate given the current implementation.

We do claim:

  • Commits are hashed client-side before transmission
  • Agents sign the full commit envelope with their own Ed25519 key
  • Records are verifiable offline without DarkMatter servers
  • Missing integrity fields fail verification by default
  • Canonicalization is cross-language specified and test-vector backed
  • Each checkpoint is co-signed by an independently operated witness
  • Checkpoint continuity between snapshots is verifiable
  • The signing key is persistent across server restarts

We do not yet claim:

  • Witnesses independently reconstruct the Merkle tree from raw data (Phase 4B)
  • Multi-party consensus (2-of-3 witness quorum) (Phase 4B)
  • Blockchain anchoring
  • Censorship resistance

Timestamp semantics

client_timestamp is what the agent asserts at commit time. It is part of the signed envelope and authenticated by the agent signature. DarkMatter cannot alter it without breaking the signature.

accepted_at is when the DarkMatter ledger accepted the commit. It is set by the server, not alterable by the client, and used in the Merkle leaf binding.

These are deliberately separate. A commit submitted at 3pm but accepted at 3:01pm has both timestamps recorded. Auditors can distinguish between what the agent claimed and when the ledger actually recorded it.

Key lifecycle

Agent signing keys follow a defined lifecycle:

  • Registration — public key PEM registered with DarkMatter. Private key never transmitted.
  • Rotation — old key marked with valid_until. Remains valid for historical verification of past commits. New key takes over for new commits.
  • Revocation — key explicitly invalidated with a reason. Commits signed after revocation are flagged revoked_key. Commits signed before revocation are flagged revoked_post_commit — integrity unaffected.

Historical verification uses GET /api/agents/:agentId/pubkey?at=<ISO> to retrieve the key that was valid at a specific point in time. Verification after rotation always uses the correct key for the commit's accepted_at timestamp.

Integrity Spec v1.0

The complete protocol specification is published at INTEGRITY_SPEC_V1.md. It is frozen — changes require a new version number. Every commit records its spec_version.

The spec covers: canonical serialization rules, commit envelope schema, agent signatures, dual timestamps, Merkle leaf format, checkpoint format v3, proof receipt format, export bundle format v3.0, verifier behaviour, and spec versioning policy.

Test vectors

Any SDK implementation must pass all test vectors before release. Vectors are published in the GitHub repository.

  • integrity_test_vectors.json — 18 canonical serialization vectors
  • merkle_test_vectors.json — 11 Merkle tree vectors (leaf hashes, roots, proofs)
# Run test vectors against the verifier
python verify_darkmatter_chain.py dummy \
  --test-vectors integrity_test_vectors.json

python verify_darkmatter_chain.py dummy \
  --test-vectors-merkle merkle_test_vectors.json
