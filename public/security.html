Security posture

What we secure.
What you can verify.

DarkMatter is not a system you trust. It is a system you can verify. This page describes how we secure your records and what controls are in place — without overclaiming.

Records can be verified independently, without relying on DarkMatter.

Infrastructure

Hosted on Railway + Supabase

API runs on Railway (Node.js/Express). Verifiable execution records are stored in Supabase Postgres with row-level security — no cross-account data access is possible through the API. Self-hosting is available under the MIT license for full infrastructure control.

Cryptographic integrity

Client-side hashing. Agent-signed records.

Payload hashes are computed client-side before transmission. Agents sign the full commit envelope with their own Ed25519 key. Every record receives a Merkle inclusion proof. Tampering is detectable without trusting DarkMatter — verification requires no trust in us or you.

Transport & access

TLS in transit. RLS in storage.

All API traffic is TLS 1.2+. Authentication uses Supabase JWT or agent API keys. Row-level security enforces account isolation at the database layer — API keys scope access to the owning agent only.

Certification status

Honest about what we have — and don't.

SOC 2 audit period is expected Q1 2027. We do not hold SOC 2, ISO 27001, or HIPAA BAA at this time. Security questionnaires and current controls documentation are available on request.

For regulated deployments, the cryptographic verification model — client-side hashing, agent signatures, offline verification — provides evidence of record integrity that does not depend on our certification status.
